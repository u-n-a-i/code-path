# Resolución DNS

Los seres humanos recordamos nombres, no números. Recordamos `google.com`, no `142.250.189.206`. El Sistema de Nombres de Dominio (DNS) resuelve esta discrepancia cognitiva traduciendo nombres legibles a direcciones IP enrutable que las máquinas pueden utilizar para establecer conexiones. Esta traducción no ocurre mágicamente: requiere que cada sistema esté configurado con servidores de nombres confiables, que las consultas sigan protocolos estandarizados y que las respuestas se cacheen inteligentemente para equilibrar precisión con rendimiento. Cuando la resolución DNS falla, la conectividad se rompe incluso si la red física funciona perfectamente: el sistema no sabe hacia dónde dirigir el tráfico porque no puede interpretar el destino nominal.

## Archivo `/etc/resolv.conf` y systemd-resolved

El archivo `/etc/resolv.conf` es el punto de configuración local que indica al resolvedor del sistema qué servidores DNS consultar para traducir nombres a direcciones. Su formato es simple: directivas `nameserver` seguidas de direcciones IP, opcionalmente acompañadas de `search` para dominios de búsqueda y `options` para ajustar comportamiento.

```
# Generated by systemd-resolved
nameserver 127.0.0.53
options edns0 trust-ad
search ejemplo.local
```

En sistemas modernos con systemd, este archivo típicamente apunta al resolvedor local (`127.0.0.53`), que actúa como proxy hacia servidores DNS upstream configurados mediante NetworkManager, Netplan o archivos de configuración específicos. Esta arquitectura en capas permite caching local, filtrado de consultas y gestión dinámica de servidores según la red activa (Wi-Fi doméstico vs. corporativo vs. móvil).

La gestión automática de `/etc/resolv.conf` puede generar confusión durante el diagnóstico. En Ubuntu, el archivo suele ser un enlace simbólico a `/run/systemd/resolve/stub-resolv.conf`, regenerado en cada arranque o cambio de red. Editarlo manualmente produce cambios efímeros que se pierden al reiniciar el servicio o el sistema. La configuración persistente debe aplicarse en la fuente: `systemd-resolve --set-dns` para cambios temporales, Netplan o NetworkManager para cambios permanentes.

> El resolvedor consulta los servidores en el orden listado en `/etc/resolv.conf`. Si el primero no responde dentro del timeout (típicamente 5 segundos), prueba el siguiente. Configurar múltiples servidores mejora la resiliencia, pero también puede enmascarar problemas: un servidor lento o con datos desactualizados puede causar retrasos intermitentes difíciles de diagnosticar.

## Uso de `dig` y `nslookup`

Estas dos herramientas permiten consultar servidores DNS directamente, bypassing el resolvedor local para diagnóstico preciso. Aunque comparten propósito, difieren en filosofía, salida y mantenimiento.

`dig` (*Domain Information Groper*) produce información estructurada y detallada. Una consulta típica como `dig ejemplo.com` revela: cabecera con código de respuesta (NOERROR, NXDOMAIN), sección de pregunta replicando la consulta, sección de respuesta con registros A/AAAA, metadatos del servidor consultado, tiempo de respuesta y estadísticas de transferencia. La opción `+short` produce salida minimalista (solo la dirección IP), útil para scripting; `+trace` sigue la cadena de delegación desde la raíz hasta el servidor autoritativo, revelando la jerarquía completa de resolución.

```bash
$ dig ejemplo.com +short
93.184.216.34

$ dig ejemplo.com MX +short
0 .
```

`nslookup` ofrece una interfaz interactiva que permite cambiar de servidor DNS en tiempo real sin reiniciar el comando. Dentro de la sesión, `server 8.8.8.8` redirige consultas al DNS público de Google; `set type=MX` cambia el tipo de registro consultado. Aunque funcional, su formato de salida es menos consistente que `dig`, y su desarrollo está estancado: `dig` es la herramienta recomendada para diagnóstico profesional.

Ambas herramientas respetan la configuración local por defecto, pero permiten sobrescribirla consultando servidores específicos: `dig @1.1.1.1 ejemplo.com` evade el resolvedor configurado para probar un DNS alternativo. Esta capacidad es esencial para aislar problemas: si una consulta falla con el servidor local pero funciona con `8.8.8.8`, el problema reside en la configuración o conectividad del resolvedor, no en la autoridad DNS del dominio.

> Las respuestas DNS pueden variar según el servidor consultado debido a caching, geolocalización, filtrado o políticas de privacidad. Comparar resultados entre `dig @local`, `dig @1.1.1.1` y `dig @autoritativo` revela discrepancias que ayudan a diagnosticar problemas de propagación, envenenamiento de caché o bloqueo regional.

## Quédate con...

*   `/etc/resolv.conf` configura los servidores DNS que el sistema consulta; en Ubuntu moderno, típicamente apunta al resolvedor local (`127.0.0.53`) gestionado por systemd-resolved.
*   La edición manual de `/etc/resolv.conf` es efímera en sistemas con systemd; la configuración persistente debe aplicarse en Netplan, NetworkManager o mediante `systemd-resolve`.
*   `dig` proporciona consultas DNS detalladas y estructuradas, con opciones como `+short` para scripting y `+trace` para seguir la cadena de delegación desde la raíz.
*   `nslookup` ofrece interfaz interactiva para cambiar servidores y tipos de registro en tiempo real, pero su formato es menos consistente y su desarrollo está estancado.
*   Consultar servidores específicos con `@ip` permite aislar problemas: si falla con el resolvedor local pero funciona con un DNS público, el problema es de configuración local, no de autoridad DNS.
*   Las respuestas DNS pueden variar entre servidores por caching, geolocalización o políticas; comparar múltiples fuentes ayuda a diagnosticar discrepancias o manipulaciones.
